# Session Briefing Template (agent variant)

Canonical structure for `SESSION_BRIEFING.md` — markdown, one per project, in the
session-briefings hub. Every numbered section must be present in every version; only §3
is replaced on update (the rest carry forward). `scripts/briefing.py check` enforces this.

Operational reference (build/run/conventions/env) belongs in the project's agent context file
(AGENTS.md / CLAUDE.md / GEMINI.md), not here. Personal working style belongs in file-based memory,
not here. This briefing holds only *timestamped* project state.

## Header

```
# <PROJECT> — Session Briefing

**v<X.Y> — <YYYY-MM-DD>** · _agent-session-briefing_
<!-- meta: parent=<umbrella or none> | layer=<optional tag, e.g. OPEN-MIT | COMMERCIAL | CLOSED | unset> | projectdir=<~/path — auto-stamped by `new`> | status=<one-line headline> -->
Operational reference lives in the agent context file (AGENTS.md / CLAUDE.md); this briefing holds state, decisions, and trajectory.
```

`bump` maintains the version and date — don't edit them by hand.

## Sections

### §1 — Project Context
One or two lines plus the current phase (design / build / testing / shipped). The full "what and
how" is in the context file; don't duplicate it. Update only on a genuine phase change.

### §2 — Current Status
A table of every workstream and its status. Use only: `COMPLETE` / `IN PROGRESS` / `NOT STARTED` /
`PENDING` / `BLOCKED` / `DEFERRED` / `IN REVIEW` (optionally tagged `UPDATED vX.Y`). No vague
phrasing — glanceable state is the point. `check` warns on out-of-vocabulary tokens.

### §3 — This Session — What Was Done
The only section replaced wholesale each update. Sub-bullets per distinct piece of work. Reference
commit SHAs instead of re-narrating file diffs; capture the *why*, decisions made, and downstream
effects that rippled. Be specific: "wired calendar-trigger notification with a fixed identifier so
each call replaces the prior schedule" beats "updated settings."

### §4 — Next Steps / Priorities
Ordered by dependency. Note blockers and prerequisites.

### §5 — Design Decisions Log
Append-only. The rationale behind locked choices — the "do not relitigate" record. Never remove
entries; stamp the version each one entered.

### §6 — Open Questions
Unresolved questions, each with the point at which it blocks progress. Move an entry to §5 once
it's decided.

### §7 — Discrepancies (spec vs code)
Where specs/docs diverge from the code. Code is authoritative unless stated; verify against the
source. Name both sources and the verdict. Carry until resolved. "None identified." when clean.

### §8 — Session Opening Prompt
A short paste-to-resume payload. Keep it lean — the context file auto-loads the operational
context, so this points the next session at current state and priorities rather than re-explaining
the project.

### §9 — Version History
One row per version (version · date · one-line summary). Append-only. Git holds the full diff
history, so this stays short.

## Constellation & surfacing (multi-segment projects)

When a project has sub-products, each segment is its own briefing and a parent briefing aggregates them:

- A child declares its parent (+ optional layer) in the header meta line:
  `<!-- meta: parent=acme-platform | layer=OPEN-MIT | projectdir=~/code/acme-api | status=<headline> -->`.
- The parent's §2 carries a script-managed **constellation block** (between `constellation:start` /
  `constellation:end` markers) — a component matrix regenerated by `rollup`, never hand-edited.
- To bubble a blocker or decision from a child up to the parent, add a clean standalone `[surface] <note>` line (prefix form, one line, never inside a table cell) in §4/§5/§6 — i.e.
  `[surface]`. It appears under "Surfaced from components" in the parent until you untag it.

## Visual references
Link mockups / screenshots / diagrams from `refs/` rather than embedding them — e.g.
"§3: adopted the dark variant — see `refs/home-screen-v2.html`." Keeps the markdown diffable.
